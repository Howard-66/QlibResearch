# qlib 工作流节点解读指南

## 1. 文档目的

本文档用于解释 `notebooks/qlib_workflow_by_code_explicit.ipynb` 各个节点的输出应该如何阅读、如何判断，以及不同指标在研究中的实际意义。

它不是某一次单独运行结果的复盘，也不是某一组固定参数的结论，而是一份可重复使用的“读数手册”。

如果后续你切换了：

- 股票池，例如 `watchlist / csi300 / csi500 / merged_csi300_500`
- 特征组
- 训练窗口和验证窗口
- 最近窗口回测与长期 walk-forward 开关

这份文档仍然适用。

---

## 2. 先理解这套工作流在回答什么问题

这份 notebook 本质上在回答四个问题：

1. 当前数据和股票池是否干净、覆盖是否足够。
2. 当前特征是否足以支持一个稳定的横截面排序模型。
3. 模型在“最新一期”会把哪些股票排在前面。
4. 如果在历史上反复按同样方式训练、预测、调仓，这个模型到底有没有持续的选股能力。

因此，解读时不要只盯着“最新 Top10 是谁”或“总收益是多少”。

更合理的顺序通常是：

1. 先看股票池和 panel 是否可信。
2. 再看特征和覆盖是否可信。
3. 再看滚动回测和长期 walk-forward 是否可信。
4. 最后再看最新一期预测名单。

---

## 3. 节点解读指南

### 3.1 节点 1：目标与官方 Workflow 映射

这一节不负责判断模型好坏，主要负责确认“这份 notebook 覆盖了哪些研究环节”。

通常它会把流程拆成：

1. Data Prepare
2. Feature Prepare
3. Train
4. Predict
5. Backtest
6. Publish

这一节的正确读法是：

- 它是流程地图，不是结果页。
- 如果后面某一节结果异常，先回到这里判断异常属于哪一类问题。
- 如果 notebook 缺少 `Backtest` 或 `Publish` 对应节点，那么它更像是实验草稿，而不是完整研究流程。

### 3.2 节点 2：环境检查与运行时初始化

这一节的作用是验证“实验环境是否可信”。

重点看：

- `project_root`
- `cwd`
- `python_version`
- `finance_data_hub_version`
- `pyqlib_version`
- `qlib_runtime_initialized`

这一节的正确读法是：

- 如果 `pyqlib` 或数据依赖缺失，后面训练和回测都不值得解读。
- `qlib_runtime_initialized=False` 不一定是错误，通常只是说明这一步还没真正开始训练。
- 如果路径不对，例如 `cwd` 指到错误目录，后面的 panel 文件和模型快照很可能读错。

### 3.3 节点 3：参数区

这一节决定“这次实验到底在测什么”。

解读时至少要先把这些参数翻译成自然语言：

- `UNIVERSE_PROFILE`
- `ACTIVE_FEATURE_GROUPS`
- `TRAIN_WEEKS`
- `VALID_WEEKS`
- `EVAL_COUNT`
- `STEP_WEEKS`
- `TOPK`
- `INDUSTRY_NORMALIZATION`
- `LONG_WALK_FORWARD_ENABLED`

例如：

- `UNIVERSE_PROFILE=csi300` 可以翻译成“在 CSI300 成分股范围内选股”
- `TOPK=10` 可以翻译成“每次只买分数最高的 10 只”
- `TRAIN_WEEKS=260, VALID_WEEKS=52` 可以翻译成“每次用约 5 年训练、1 年验证”

这一节最常见的误读是：

- 没先讲参数，就直接解读收益
- 把最近 1 年窗口结果，当成长期稳健性结论
- 把 `LONG_WALK_FORWARD_EVAL_COUNT=0` 误解成“不跑长期评估”

通常 `eval_count=0` 更接近“用指定日期区间内所有可评估点”。

### 3.4 节点 4：Universe 解析与 Panel 导出

这一节先看股票池口径，再看样本范围。

重点输出包括：

- `rows`
- `instrument_count`
- `start_date`
- `end_date`
- `csi300_rows`
- `csi500_rows`
- `csi300_instruments`
- `csi500_instruments`

这一节有两个最重要的概念。

第一个概念是：`instrument_count` 是整个时间区间里的历史唯一股票数，不是当前时点股票数。

例如在严格 `csi300` 口径下，`instrument_count=607` 通常表示“2016-2026 期间曾经进入 CSI300 的唯一股票有 607 只”，并不表示“当前有 607 只成分股”。

第二个概念是：`rows` 与 `csi300_rows` 的关系，直接决定股票池过滤是否真的进了主流程。

解读方法：

- 如果你跑的是严格 `csi300`，理想状态通常是 `rows == csi300_rows`
- 如果 `rows` 明显大于 `csi300_rows`，说明 panel 里仍混有非目标成分股样本
- 如果预览表里 `in_csi300=False` 频繁出现，而这次实验又声称是 `csi300`，通常要优先怀疑股票池口径没有真正收紧

这一节还要理解标签字段：

```text
future_return_4w = 未来4周绝对收益
label_excess_return_4w = future_return_4w - 同期股票池平均future_return_4w
```

因此，这个模型训练的不是“未来会不会涨”，而是“未来相对同池平均水平会不会更强”。

### 3.5 节点 5：特征组选择、行业标准化与 Panel EDA

这一节的作用是回答三个问题：

1. 特征够不够全。
2. 特征够不够干净。
3. 数据覆盖够不够稳定。

重点输出通常有：

- `selected_feature_count`
- `filtered_feature_count`
- `removed_feature_count`
- panel 覆盖图
- 缺失率图
- 高相关候选表

这一节的正确读法是：

- `selected_feature_count` 是你希望使用的特征数
- `filtered_feature_count` 是真正进入后续训练的可用特征数
- `removed_feature_count` 高，不一定代表模型差，也可能只是某些字段在当前股票池里长期缺失

常见判断方法：

- 如果大部分保留特征的缺失率在 `0%~10%`，一般算比较健康
- 如果很多核心特征缺失率超过 `30%~40%`，模型结果要打折
- 如果覆盖图在评估区间内长期接近目标股票池上限，说明数据可用性高

例如：

- 严格 `csi300` 下，panel 每周覆盖在 `266~300` 之间、均值接近 `290`，通常可读作“样本池整体稳定，少量缺口来自个别成分股数据不足”
- 如果滚动回测时 `coverage_mean` 还能接近 `299`，说明真正能打分的股票几乎覆盖整个股票池

高相关特征的正确读法是：

- 高相关不代表错误
- 它代表“这几个特征在表达相似信息”
- 在树模型里可以容忍一定相关性，但如果高相关对过多，特征解释性会下降

### 3.6 节点 6：显式训练节点

这一节主要是“训练诊断”，不是最终证据。

重点看：

- `latest_feature_date`
- `used_feature_count`
- `train / valid / test` 时间切分
- `feature_importance`

这里最容易误读的地方有两个。

第一，这一节通常是“单次训练切分”，它只是为了显式展示训练对象如何构造；它不等于滚动回测。

第二，`feature_importance` 只能解释“这次训练里模型经常使用了哪些特征”，不能直接当成因果结论。

正确的理解方式是：

- 如果 `used_feature_count` 很低，先检查特征缺失和预过滤
- 如果训练/验证区间太短或明显不合理，后面结果要谨慎
- 如果重要性高度集中在个别技术变量，说明模型可能更偏“时点行情解释”，而不是稳定的价值排序

例如：

- 若 `macd_hist`、`rsi`、`pb`、`ni_ttm` 排在前列，可读作“模型更依赖技术强弱、估值水平和盈利能力的混合信息”

### 3.7 节点 7：显式预测节点

这一节看的是“最新一期模型把谁排在前面”，但它不是证明模型有效的核心证据。

重点看：

- `prediction_count`
- `top_score`
- 最新 Top20 的行业分布
- 最新 Top20 的 `in_csi300 / in_csi500`

正确读法：

- `prediction_count` 应尽量接近目标股票池规模
- 如果是严格 `csi300`，最新 Top20 基本都应当 `in_csi300=True`
- 分数绝对值的大小不要过度解读，尤其在大盘股池里，分数经常会被压缩到 0 附近

一个很重要的经验是：

- 这类模型更适合用“分数排序”来解释
- 不适合把分数直接当成未来精确收益率预测

所以即使 Top10 中有很多负分，也不一定表示“模型看空它们”，更可能只是“在同池比较中，它们仍然排在更靠前的位置”。

### 3.8 节点 8：显式滚动回测节点

这是整份 workflow 最关键的结果节点。

它回答的是：

“如果每个评估点都重新训练，再按同样方式选股，这套方法历史上到底有没有持续的排序能力和组合收益能力？”

通常这一节至少要分成两层来读。

第一层是“快速滚动评估”，例如最近 1 年窗口。

它更适合回答：

- 模型在当前市场风格里顺不顺
- 最近一段时间是否表现偏强

第二层是“长期 walk-forward”。

它更适合回答：

- 模型跨年份是否还成立
- 最近窗口的亮眼表现是不是只是一段顺风期

对长期 walk-forward 的一个常见误解是：

- 看到总收益更高，就以为长期结果更好

其实更合理的对比方式是：

- 比较 `RankIC_IR`
- 比较 `topk_mean_excess_return_4w`
- 比较 `CAGR`
- 比较 `strategy_max_drawdown`
- 比较 `Sharpe`

例如：

- 如果最近 1 年 `CAGR` 很高，但长期 `CAGR` 明显回落，同时长期 `max_drawdown` 更深，那么更合理的结论是“模型长期有效，但最近 1 年偏强”
- 如果长期 `RankIC_IR` 仍为正，而长期 `Sharpe` 一般，通常可读作“排序信号存在，但转成组合收益后的强度有限”

### 3.9 节点 9：分析与可视化

这一节不是新增结论，而是验证第 8 节的结论是否真的站得住。

通常要重点看四类图：

1. `窗口结果对比`
2. `RankIC / TopK 超额收益时间序列`
3. `净值曲线`
4. `月度 / 年度收益热力图`

`窗口结果对比` 的作用是把“最近窗口”和“长期窗口”并排放在一起，方便判断当前结果有没有被某个局部时期放大。

`RankIC / TopK 超额收益时间序列` 的作用是看稳定性。

正确读法：

- 均值为正不够，还要看正值出现得频不频繁
- 如果 `RankIC` 正值周数明显多于负值周数，排序信号更可信
- 如果 `TopK 超额收益` 只有少数几次暴涨拉高均值，策略稳定性要打折

`净值曲线` 的作用是看路径，而不是只看终点。

重点关注：

- 是否经常创新高
- 回撤是否长期修复不了
- 短窗口和长窗口曲线风格是否一致

`月度 / 年度收益热力图` 的作用是看收益是否均匀分布。

例如：

- 如果某个模型 4 年总收益不错，但 80% 的收益都来自 1 个年份，那么它的稳健性仍然需要谨慎
- 如果连续多年年度收益为正，即使幅度不极端，也往往比“某一年特别亮眼、其他年份很平”更有研究价值

### 3.10 节点 10：可选发布与产物落盘

这一节不是用来证明模型强弱的，而是用来保证结果可复盘、可交付。

通常应重点关注：

- 是否落盘了 `feature_prefilter.csv`
- 是否落盘了 `feature_importance.csv`
- 是否落盘了 `latest_prediction.csv`
- 是否落盘了 `rolling_summary.csv`
- 是否落盘了 `rolling_details.csv`
- 是否落盘了 `equity_curve.csv`

如果一个结果看起来很好，但没有把这些中间产物保存下来，后续就很难判断它到底是：

- 数据口径变化造成的
- 特征变化造成的
- 滚动窗口变化造成的
- 还是偶然结果

## 4. 专业指标速查

### 4.1 RankIC

定义：

- 模型分数排序与未来超额收益排序的 Spearman 相关系数

含义：

- 大于 0，说明高分股票整体更容易排在未来表现更好的位置

如何判断：

- 长期均值稳定为正，就说明有横截面排序信息
- 周频横截面里，`0.02~0.05` 往往就已经有研究价值

### 4.2 RankIC_IR

定义：

- `RankIC` 的均值除以标准差

含义：

- 它衡量的不是“强不强”，而是“稳不稳”

如何判断：

- `> 0`：有信号，但不一定稳
- `~ 0.2`：通常可以认为有一定稳定性
- `> 0.3`：更让人放心

以上只是经验阈值，不是硬规则。

### 4.3 IC

定义：

- 模型分数与未来超额收益的 Pearson 线性相关系数

含义：

- 它更看重“分数大小”和“未来收益大小”是否成比例

如何判断：

- `RankIC` 为正、`IC` 近 0，很常见
- 这通常意味着模型更适合做排序，不适合把分数直接当收益预测值

### 4.4 topk_mean_return_4w

定义：

- 每个评估点里，TopK 股票未来 4 周的平均绝对收益

如何判断：

- 它反映头部股票“赚没赚钱”
- 但它不告诉你“是否跑赢同池平均”

### 4.5 topk_mean_excess_return_4w

定义：

- 每个评估点里，TopK 股票未来 4 周相对同池平均收益的超额收益

如何判断：

- 这是比 `topk_mean_return_4w` 更重要的 alpha 指标
- 长期稳定为正，说明头部 picks 不是只靠市场整体上涨在赚钱

### 4.6 topk_hit_rate

定义：

- TopK 股票里未来绝对收益大于 0 的比例

如何判断：

- 不要求必须大于 50% 才能赚钱
- 如果命中率不高但超额收益仍为正，通常表示“少数胜利贡献较大”

### 4.7 coverage_mean

定义：

- 每个评估点里，真正得到有效分数的股票数均值

如何判断：

- 越接近目标股票池规模越好
- 如果长期明显偏低，说明样本缺失或特征可用性不足

### 4.8 strategy_total_return / CAGR / Max Drawdown / Sharpe

这些是组合层面的结果，不是信号层面的结果。

正确理解方式：

- `strategy_total_return` 看终局收益
- `CAGR` 看年化强度
- `Max Drawdown` 看过程痛苦程度
- `Sharpe` 看风险调整后的效率

其中：

- 长窗口比较时，不要只看总收益，要优先看 `CAGR` 和 `Sharpe`
- 如果 `Sharpe` 不高但 `RankIC_IR` 为正，往往说明“信号存在，但交易转化效率一般”

## 5. 回测假设与限制

当前这套 workflow 的组合回测有几个必须明确的假设：

1. 没有显式计入手续费。
2. 没有显式计入滑点。
3. 没有显式计入冲击成本。
4. 没有显式处理涨跌停和真实撮合限制。
5. 调仓使用的是简单价格输入，不是严格的开盘成交模拟。
6. 采用等权 TopK 组合。
7. 带有简化版止损逻辑。

这意味着：

- 当前结果更接近“成本前、理想化回测”
- 如果未来策略换手高，真实净收益往往会低于这里看到的毛收益
- 因为存在简化止损，所以不能把它当成“无风控纯信号组合”的回测结果

因此，一个比较成熟的解读顺序应该是：

1. 先看这套无成本回测是否至少有正的长期信号
2. 再把手续费、滑点和换手加入后重新评估
3. 最后再讨论是否具备实盘价值

## 6. 常见组合判断模板

### 6.1 如果最近窗口强、长期窗口弱

更合理的结论通常是：

- 模型最近适应当前市场
- 但长期稳健性一般
- 不能把最近的高收益直接外推

### 6.2 如果长期 RankIC 为正，但 Sharpe 一般

更合理的结论通常是：

- 排序信号存在
- 但交易转化效率一般
- 组合层面可能受换手、调仓方式、风控方式影响较大

### 6.3 如果 TopK 命中率低于 50%，但超额收益仍为正

更合理的结论通常是：

- 策略并不是“赢得更多次”
- 而是“赢的时候赚得更多”

### 6.4 如果 coverage_mean 明显低于股票池规模

优先怀疑：

1. 特征缺失过多
2. 股票池过滤口径异常
3. 预处理后丢掉了太多样本
4. 某些字段只在部分股票上可用

### 6.5 如果最新预测分数都很小

不要立刻判断模型无效。

更常见的解释是：

- 当前股票池更同质
- 标签本身是超额收益，天然围绕 0 波动
- 模型更适合做相对排序，而不是做绝对收益预测

## 7. 阅读一个新结果时的推荐顺序

建议每次都按下面顺序读：

1. 节点 3：先把参数翻译成自然语言。
2. 节点 4：确认股票池和标签定义没问题。
3. 节点 5：确认 panel 覆盖、缺失率和高相关特征情况。
4. 节点 8：先看长期 walk-forward，再看最近窗口。
5. 节点 9：用时间序列、净值曲线和热力图验证第 8 节结论。
6. 节点 6、7：最后再解释模型学了什么、最新一期押了什么。

如果只按这条顺序做，你通常就能避开这类工作流里最常见的三个误区：

1. 只看最近 Top10，不看长期稳健性。
2. 只看总收益，不看股票池口径和 panel 覆盖。
3. 只看短窗口亮眼结果，不看长期 walk-forward 的“降温作用”。

## 8. 相关代码位置

如果需要从实现层面核对定义，可以优先看这些文件：

- `src/ml/weekly_feature_panel.py`：标签与 panel 构建
- `src/ml/qlib_pipeline.py`：特征组、行业标准化、训练输入
- `src/ml/weekly_model_eval.py`：滚动评估、指标汇总、信号矩阵
